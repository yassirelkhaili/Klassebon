1 — Projektantrag
1.1 Kurzform der Aufgabenstellung
Projektname KlasseBon
Aufgabe Entwicklung einer webbasierten Anwendung, die es Privatpersonen ermöglicht, ihre Abonnements und Quittungen digital zu dokumentieren, um eine klare Übersicht über ihre monatlichen Ausgaben zu erhalten. Quittungen können per OCR (Tesseract) automatisch eingelesen werden. Zusätzlich analysiert das System die erfassten Kosten und generiert personalisierte Spartipps über ein lokales KI-Modell (Ollama).
Kernfunktionen
  • Benutzerregistrierung und Login
  • Erfassung von Abonnements und einzelnen Ausgaben
  • Automatische Quittungserkennung per OCR (Tesseract)
  • Übersicht der monatlichen Gesamtkosten
  • Kategorisierung der Ausgaben
  • Generierung von Spartipps über ein lokales LLM (Ollama)
1.2 Ist-Analyse

Aktuell verwalten viele Privatpersonen ihre Ausgaben unsystematisch. Quittungen werden häufig gesammelt, verloren oder nicht ausgewertet. Abonnements laufen im Hintergrund weiter, ohne dass ein klarer Überblick über die Gesamtkosten besteht.

Problem Beschreibung
Fehlende Transparenz Kein Überblick über monatliche Ausgaben
Unterschätzte Kosten Kleine, aber regelmäßige Kosten werden unterschätzt (z. B. Streaming-Abos)
Keine Auswertung Keine strukturierte Auswertung der Ausgaben
Kein Sparpotenzial Kaum konkrete Hinweise auf Einsparpotenziale
Marktvergleich
Produkt PSD2 / Multi-Banking Auto-Kategorisierung Analysefunktionen Besonderheiten
Finanzguru Ja, Kontenverknüpfung, Multibanking Ja, KI-gestützt Detaillierte Analysen, Vertragsüberblick Vertragsoptimierung, Sparvorschläge
MoneyControl Teilweise Kontenverwaltung Teilweise, für wiederkehrende Buchungen Budget, Vorschau, grafische Auswertungen Klassisches Haushaltsbuch, viele Auswertungen
Outbank Ja, Multi-Banking Ja, nach Kategorien/Tags Auswertungen, Budgets Offline-Konten, Haushaltsbuch-Funktion
KlasseBon Nein, bewusst keine Bankanbindung Einfache Kategorisierung + OCR-Erkennung (Tesseract) Übersicht monatlicher Gesamtkosten, Spartipps via lokalem LLM (Ollama) Schlanke Web-App, OCR-Quittungserkennung, lokale KI-Spartipps, Docker-basiert, Fokus auf Transparenz & Datenschutz

Für viele Nutzer ist eine schlanke, webbasierte Anwendung sinnvoll, bei der Ausgaben manuell erfasst und bewusst reflektiert werden. Genau hier setzt „KlasseBon" an: Fokus auf Übersichtlichkeit, einfache Bedienung und gezielte Spartipps.

2.2 Welche Anforderungen müssen erfüllt sein?
Kategorie Anforderung
Funktional (Kern)
  • Benutzerregistrierung, Login und Passwort-Reset über Better-Auth (Session-basierte Authentifizierung)
  • Strukturierte Erfassung von Ausgaben/Abonnements (Titel, Kategorie, Betrag, Datum, Wiederkehrstatus)
  • Automatische Quittungserkennung per OCR (Tesseract) — Bild hochladen, Betrag & Kategorie werden extrahiert
  • Dashboard mit monatlicher Gesamtkostenübersicht (Gesamtsumme + Balkendiagramm pro Kategorie)
  • Vordefinierte Kategorien (Streaming, Lebensmittel, Versicherung, Transport, Sonstiges)
  • Spartipp-Generierung über lokales LLM (Ollama) — z. B. „Ihre Streaming-Ausgaben (45€) überschreiten 30% Ihres Budgets"
  • Vollständige CRUD-Operationen für Ausgaben und Abonnements
Fehlerquote < 1% aller Requests durch strukturiertes Error Handling und Logging (Winston)
Testabdeckung ≥ 80% Unit Test Coverage (nachgewiesen durch Jest/Vitest Coverage Report)
Performance API-Response < 500ms (95th Percentile), Frontend-Ladezeit < 2s (Lighthouse Score ≥ 90)
Sicherheit
  • Passwort-Hashing mit scrypt (Better-Auth Default, memory-hard)
  • Session-basierte Authentifizierung mit DB-Storage (Better-Auth)
  • CSRF-Schutz (Better-Auth eingebaut)
  • Rate Limiting (Better-Auth eingebaut + max. 100 Requests/Min pro IP)
  • Input Validation mit Zod
Benutzerfreundlichkeit Responsive Design (Mobile, Tablet, Desktop), jede Kernfunktion in max. 3 Klicks erreichbar
Wartbarkeit Einheitlicher Codestil durch ESLint & Prettier
2.3 Welche Einschränkungen müssen berücksichtigt werden?
Einschränkung Beschreibung
Bankanbindung Keine Integration externer Bank-Schnittstellen (PSD2)
Datenschutz Datenschutzanforderungen gemäß DSGVO müssen eingehalten werden
OCR-Genauigkeit Tesseract-Erkennung kann je nach Belegqualität ungenau sein — bei Fehlern ist eine manuelle Eingabe als Fallback vorgesehen.
Ollama Hardware Lokales LLM erfordert mind. 8 GB RAM (3B Modell), empfohlen 16 GB + GPU. Ohne ausreichende Hardware ist die Spartipp-Generierung eingeschränkt.
Keine externen API-Aufrufe Alle Dienste (Tesseract, Ollama) laufen lokal — keine Daten verlassen den Rechner des Nutzers.
Open Source Alle eingesetzten Technologien sind vollständig Open Source (MIT / Apache 2.0).
Zeit Projektzeitraum begrenzt auf 14 Wochen
Team Begrenzte personelle Ressourcen (4 Mitglieder)
2.4 Rechtliche Maßnahmen (DSGVO)
Maßnahme Beschreibung
Einwilligung (Art. 6 Abs. 1 lit. a) Klarer Zustimmungsdialog beim ersten Start der App
Vertragserfüllung (Art. 6 Abs. 1 lit. b) Verarbeitung personenbezogener Daten zur Vertragserfüllung
Datenschutzerklärung Verständliche Erklärung in einfachem Deutsch, was gesammelt wird und warum
Recht auf Löschung (Art. 17) Nutzer können ihr Konto und alle zugehörigen Daten jederzeit löschen (Recht auf Vergessenwerden)
Datensparsamkeit (Art. 5 Abs. 1 lit. c) Es werden nur die für die Funktion unbedingt notwendigen Daten erhoben
Widerrufsmöglichkeit Die Einwilligung kann jederzeit so einfach widerrufen werden, wie sie erteilt wurde
Drittlandübermittlung (Art. 44) Entfällt — Spartipp-Generierung erfolgt lokal über Ollama (Self-Hosted LLM). Es werden keine personenbezogenen Daten an Drittländer übermittelt.
2.5 Technische Sicherheitsmaßnahmen
Maßnahme Beschreibung
Passwortsicherheit Passwörter werden mit scrypt gehasht (Better-Auth Default, memory-hard Algorithmus) — Klartext-Passwörter werden niemals gespeichert
Verschlüsselte Datenbank Finanzdaten in verschlüsselter PostgreSQL-Datenbank gespeichert
Session-Sicherheit Session-basierte Authentifizierung via Better-Auth — Sessions in der Datenbank gespeichert, automatische Verlängerung, serverseitige Invalidierung möglich
Datenschutzkonformes Logging Protokollierung ohne personenbezogene Daten — keine IP-Adressen oder Nutzungsprofile
LLM-Sicherheit Ollama läuft lokal als Self-Hosted-Dienst — keine externen API-Keys, keine Datenübertragung an Dritte. Zugriff nur über internes Netzwerk / Docker-Netzwerk
